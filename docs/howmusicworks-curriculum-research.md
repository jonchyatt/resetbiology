# HowMusicWorks.org Curriculum Research

**Date:** 2026-04-03
**Source:** https://www.howmusicworks.org
**Purpose:** Map the full curriculum for integration into Reset Biology music education platform

---

## Overview

How Music Works (by ChordWizard Software) is a comprehensive music theory curriculum delivered as a free online tutorial. The site advertises **115+ topics** across **9 tutorials**, illustrated with **360 diagrams** and **750 demonstration sounds**. A desktop version (ChordWizard Music Theory 3.0) includes all content with search, bookmarks, and printing.

The curriculum progresses from absolute fundamentals (what is sound?) through advanced applied topics (improvisation, songwriting). It is entirely text + diagram + audio -- no video, no interactive exercises, no gamification.

## 2. Content Format Analysis

### What Each Lesson Contains

**Text:** Primary content delivery. Clear, simple language. Progressive -- each topic builds on previous ones. Written for musicians of any background level.

**Diagrams (360 total):** Static visual illustrations showing:
- Waveform diagrams (sound section)
- Piano keyboard overlays (scale/chord visualization)
- Guitar fretboard diagrams
- Staff notation examples
- Interval/degree charts
- Scale pattern diagrams
- Chord construction diagrams

**Audio (750 demonstration sounds):** Embedded audio clips demonstrating:
- Individual notes and intervals
- Scale playback (ascending/descending)
- Chord voicings
- Chord progressions
- Before/after comparisons (e.g., major vs minor)

**No interactive elements:** The site is purely read/listen -- there are no:
- Interactive exercises or quizzes
- Playable instruments
- Drag-and-drop activities
- Progress tracking
- Spaced repetition
- Mic-based interaction
- Gamification of any kind

### Teaching Methodology

1. **Bottom-up construction:** Starts from physics (sound waves) and builds concept by concept to complex applications (songwriting, improvisation)
2. **Instrument-aware:** Topics frequently show both piano and guitar representations (e.g., 1.11 Piano, 1.12 Guitar, 2.5 Guitar)
3. **Diagram-heavy:** Every concept gets visual representation. The 360 diagrams across ~102 topics means 3-4 diagrams per topic on average
4. **Audio-reinforced:** 750 sounds across ~102 topics means ~7 audio clips per topic, so learners hear every concept
5. **Progressive prerequisites:** Each section explicitly builds on prior sections. Section 6 requires understanding both scales (2, 4) and chords (3)
6. **Theory-first, application-second:** Sections 1-6 are pure theory; sections 7-9 are applied/practical
7. **No assessment:** Zero quizzes, exercises, or verification of understanding

### What's Missing (Our Opportunity)

- **No interactivity** -- passive consumption only
- **No ear training** -- audio is demonstrative, not testing
- **No pitch detection** -- no mic-based activities
- **No notation rendering** -- static images, not dynamic
- **No spaced repetition** -- no learning tracking whatsoever
- **No gamification** -- pure textbook style
- **No adaptive difficulty** -- linear progression only
- **No instrument-specific exercises** -- descriptions only, no practice routines

## 3. Integration Architecture

### Design Principles

1. **Curriculum as data layer** -- Topic definitions, prerequisites, and learning objectives are stored as structured JSON, completely separate from rendering
2. **Multiple renderers per concept** -- The same "C Major Scale" concept can render as: OSMD staff notation, Canvas game visual, piano keyboard overlay, guitar fretboard diagram
3. **HMW content as reference, not verbatim copy** -- We map their topic structure and progression logic, but create our own content, exercises, and interactive elements
4. **Pitch detection transforms passive into active** -- Every "listen to this" moment becomes "now YOU play/sing it"

### Proposed Data Model

```typescript
// Curriculum topic definition (curriculum-layer)
interface CurriculumTopic {
  id: string;                    // e.g., "sound-1.3" or "major-scale-2.4"
  hmwRef: string;                // e.g., "1.3" -- maps to HMW topic number
  section: CurriculumSection;    // parent section
  title: string;                 // "Amplitude and Frequency"
  slug: string;                  // URL-friendly: "amplitude-and-frequency"
  
  // Learning structure
  prerequisites: string[];       // topic IDs that must be completed first
  learningObjectives: string[];  // measurable outcomes
  difficulty: 1 | 2 | 3 | 4 | 5;
  estimatedMinutes: number;
  
  // Content references (NOT embedded content)
  notationRefs: NotationRef[];   // references to OSMD/MusicXML snippets
  audioRefs: AudioRef[];         // reference audio clips
  diagramRefs: DiagramRef[];     // custom diagram components
  
  // Exercise/assessment hooks
  exercises: ExerciseRef[];      // pitch detection, ear training, quiz, etc.
  
  // FSRS integration
  fsrsParams: FSRSTopicParams;   // spaced repetition scheduling parameters
}

interface CurriculumSection {
  id: string;                    // e.g., "sound-and-music"
  number: number;                // 1-9
  title: string;                 // "Sound and Music"
  topics: CurriculumTopic[];
  description: string;
}

// Notation reference -- separate from curriculum
interface NotationRef {
  type: 'osmd' | 'canvas-staff' | 'piano-keyboard' | 'guitar-fretboard';
  source: string;                // MusicXML file path, or component ID
  label: string;                 // "C Major Scale ascending"
  params: Record<string, any>;   // renderer-specific params
}

// Exercise definition
interface ExerciseRef {
  type: 'pitch-match'            // Sing/play the displayed note (pitchy/PESTO)
       | 'interval-identify'     // Hear two notes, name the interval
       | 'scale-play'            // Play a scale on mic, verify each note
       | 'chord-identify'        // Hear a chord, identify type
       | 'rhythm-tap'            // Tap a rhythm pattern
       | 'notation-read'         // See notation, play it
       | 'quiz';                 // Multiple choice knowledge check
  difficulty: 1 | 2 | 3 | 4 | 5;
  params: Record<string, any>;
}
```


### Integration with Existing Reset Biology Systems

**Pitch Defender / Pitch Game Franchise:**
- Sections 1-2 provide the theory curriculum backing the pitch games
- A student playing Pitch Defender is exercising skills from topics 1.5 (Frequency and Pitch), 1.9-1.12 (Chromatic Scale), and 2.1-2.3 (Major Scale)
- The curriculum layer can suggest "unlock" recommendations: "You're accurate on C Major notes -- try the D Major level!"
- FSRS tracks which pitch ranges/notes need more practice

**OSMD (OpenSheetMusicDisplay):**
- Sections 2, 3, 4 provide the notation content that OSMD renders
- MusicXML files for every scale, chord, and progression referenced in the curriculum
- Study overlay (from Jon's chamber choir vision) maps directly to curriculum topics
- Each OSMD snippet links back to its curriculum topic for context

**Canvas Staff Renderer:**
- Game-style rendering for Pitch Defender and exercises
- Lighter weight than OSMD for real-time game visuals
- Renders the same notes/scales but in game context
- Section 5 (Meter) drives rhythm game mechanics

**FSRS Spaced Repetition:**
- Each CurriculumTopic gets FSRS scheduling
- Review cards generated from exercises, not just reading
- Interval identification cards from Section 3
- Scale recognition cards from Section 4
- Mode identification cards from Section 8

### Implementation Phases

**Phase 1: Curriculum Data (Foundation)**
- Create JSON curriculum definition for all 9 sections, ~102 topics
- Define prerequisite graph
- Map learning objectives to exercise types
- No rendering yet -- pure data

**Phase 2: Notation Assets**
- Create MusicXML files for all scales, chords, intervals referenced
- Wire up OSMD rendering for each notation reference
- Piano keyboard component for visual mapping
- Guitar fretboard component (optional, lower priority)

**Phase 3: Exercise Engine**
- Pitch-match exercises using pitchy/PESTO
- Interval identification (hear and name)
- Scale playback verification (play C Major, system checks each note)
- Knowledge quizzes for theory topics

**Phase 4: FSRS Integration**
- Connect exercises to FSRS scheduler
- Review session generation (daily practice mix)
- Progress dashboard
- Adaptive difficulty based on FSRS ease factors

**Phase 5: Game Integration**
- Connect curriculum progress to Pitch Defender unlocks
- Pitch Defender gameplay feeds FSRS data back to curriculum
- Cross-pollination: playing the game IS studying the curriculum

## 5. Key Takeaways for Jon

1. **HMW has excellent progression logic** -- the prerequisite chain from physics to songwriting is well-structured and worth adopting
2. **Their content is passive** -- zero interactivity. This is our massive differentiator. Every HMW "listen to this" becomes our "now play/sing it"
3. **Pitch detection transforms the curriculum** -- Sections 1-4 especially become active learning with pitchy/PESTO
4. **The curriculum is a DAG, not a list** -- prerequisites create a directed acyclic graph. Some sections can be studied in parallel (e.g., Section 5 Meter doesn't require Section 3 Chords)
5. **FSRS makes this sticky** -- HMW is a reference you read once. With spaced repetition, our version becomes a practice system you return to daily
6. **Game integration is the killer feature** -- Playing Pitch Defender without knowing it's teaching you Section 2 and 4 content. Fun first, educational second.
