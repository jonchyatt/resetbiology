## 5. Acceptance criteria

- Receipts are plain objects with exactly the five enumerable fields `receiptVersion`, `recordId`, `eventId`, `digest`, and `acceptedAt`; the fixed version, canonical identity/fingerprint, primitive bounds, millisecond UTC round-trip, and 1,024-byte total ceiling are enforced before deletion.
- `start()` installs only online, visible visibility-change, and visible focus edges. Append notification and explicit flush share the same path. No automatic work begins hidden or offline.
- One coordinator permits one in-memory request. Concurrent triggers coalesce, and a trigger at the final empty check may request one bounded trailing pass.
- Each run acknowledges at most 25 heads. Each head gets at most three attempts: immediate, 250 ms, and 1,000 ms. Each request times out at 10,000 ms. No timing value persists.
- Every response body is stream-counted to 4,096 bytes. Fatal UTF-8 decoding and JSON parsing apply only to HTTP 200 and the known application responses at 400, 401, 403, 409, 500, and 503. Transport statuses 408, 425, 429, 502, and 504 may carry empty or malformed bodies at or below the byte ceiling and still retry.
- HTTP 200 valid receipt acknowledges; bad HTTP 200 evidence blocks corrupt. Known application responses require exact one-field JSON: 400 `invalid_event` blocks invalid, 401 `unauthorized` pauses, 403 `verify_email` or `account_mismatch` pauses, 409 `replay_conflict` blocks conflict, 500 `ledger_corrupt` blocks corrupt, and 503 `identity_lookup_failed` or `ledger_unavailable` retries. Body-agnostic transport statuses are exactly 408, 425, 429, 502, and 504 and retry only when the bounded body is at most 4,096 bytes. Every other status/body pairing, including wrong/empty/malformed application JSON, 501, and unlisted 5xx, blocks as protocol failure.
- Frozen outcomes are `empty`, `saved`, `paused-auth`, `paused-foreground`, `blocked-invalid`, `blocked-conflict`, `blocked-corrupt`, `blocked-protocol`, `retry-exhausted`, `storage-blocked`, `budget-exhausted`, and `closed`; results contain counts only.
- Strict head-of-line dispatch prevents every later event and Undo from overtaking an unacknowledged or poisoned head. No skip or dead-letter move exists.
- Exact duplicate POSTs across tabs are allowed. Identical settlement converges on one receipt; a changed receipt is corruption. No cross-tab single-request claim is made.
- Closing account A aborts A. A late response cannot touch B; A remains pending unless another A tab already committed the identical receipt.
- Quota, blocked upgrade, version change, unavailable storage, transaction abort, or receipt pruning failure cannot resolve `saved`.
- `STORAGE_CLOSED` maps to `closed`. Every other closed storage code and unknown storage exception maps to `storage-blocked`. If close happens before acknowledgement completion, `closed` wins; if the atomic acknowledgement committed first, its already-resolved `saved` remains truthful.

The deterministic suite must pass 24/24 cases covering empty, ordered success, Undo, single flight, every edge trigger, receipt hostility, the complete closed response map, exact retry delays, timeout/no-spin, lost response, 25-head budget, close/late response, account separation, and source guards. Its retry table explicitly includes both admitted 503 bodies, 408, 425, 429, 502, 504, network failure, and timeout while retaining 24 total groups.

The actual-Chrome proof must pass the existing 11 journal groups plus 18 new groups: additive migration, durable receipt reload, lost response, crash-before-settlement, transaction rollback, two-tab POST and settlement convergence, changed receipt, forged receipt matrix, retry exhaustion, invalid/conflict/corrupt poison ordering, account switch, quota, blocked/version-change/unavailable settlement, 257th receipt pruning plus old replay, trigger coverage, and fresh-page zero-residue cleanup.
