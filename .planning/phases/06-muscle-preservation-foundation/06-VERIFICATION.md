## Runtime Slice Verification Ticket

Every later Phase 6 runtime slice uses this seven-section verifier contract.

### 1. Objective

Determine whether the committed slice satisfies the original requirement and its acceptance criteria on the real production route without relying on the builder’s explanation.

### 2. Verifier Input

The blind verifier receives only:

- the controlling requirement and ticket;
- the exact candidate diff;
- the acceptance criteria and permitted write set.

The verifier does not receive builder reasoning, prior verdicts, or foreman conclusions.

### 3. Tree and Commit Gate

- Record candidate commit and `git status` before work.
- Reject a dirty tree, changed commit, unrelated diff, forbidden file, new dependency without approval, or missing runnable check.
- Run the strict build and focused tests from the committed candidate.

### 4. Isolated Write-Path Gate

- Only the named isolated relay may mutate the test account.
- Establish the baseline before the first write.
- Every created assignment or event carries a run identity.
- Read back canonical server state after every accepted write.
- Clean only objects created by the run and prove the final state equals baseline.
- Reject direct database writes or any cleanup that can touch pre-existing records.

### 5. Dual-Eye Visual Gate

- Bind every capture to candidate and production commits.
- Capture desktop and 390-pixel phone states; capture normal and reduced motion separately where motion exists.
- Eye 1 gives a blind static verdict for text, values, layout, contrast, focus, and touch geometry.
- Eye 2 gives a blind temporal verdict for countdowns, color transitions, timer behavior, pauses, completion, and transient states.
- Both verdicts must cite exact frames or timecodes before comparison.
- Audio cues require an audio-bearing recording and timecoded receipt; silent browser video is insufficient.
